# PDF-Engine
> Eine einzige Quelle für Vorschau, Download und Druck – jedes Dokument wird identisch als echtes PDF erzeugt.

## Für Anwender
Egal ob Vorschau, Herunterladen oder Drucken: immer **dasselbe** finale PDF (keine abweichende Browser-Druckansicht). Firmen-Fußzeile auf jeder Seite, „Seite X von Y", Logo, Empfänger-/Datenblock, Positionstabelle mit Gewerk-Gruppen und Summen. Viewer mit „Herunterladen/Drucken/Schließen"; **ESC schließt immer** und führt **exakt dorthin zurück, wo das PDF geöffnet wurde** – aus der Versionshistorie zurück in die wieder geöffnete Historie, aus dem Editor in den Editor, aus einer Liste/Übersicht in die Liste/Übersicht.

## Technik

**Routen & Komponenten**
- `src/components/document/printDocument.ts` – Layout-Engine: `buildHtml()`, `renderDocumentHtml()`, `printDocument()`, `printStoredHtml()`, `injectAppBar()`, Typen `PrintMeta`/`PaymentInfo`.
- `src/lib/pdf.ts` – Viewer/Steuerung: `openDocumentPdf()`, `openSnapshotPdf()`, `showPdf()`, `htmlToPdfBlob()`, `buildDocumentPdfFileName()`.
- `api/render-pdf.js` – Serverless-Renderer (PDFShift).

**Ablauf**
Im Editor öffnet das **Werkzeug-Symbol** je Position eine **Positions-Vollmaske** (`PositionEditModal`), aufgebaut wie der `ServiceEditor` mit zwei Reitern: **Informationen** (Kurz-/Langtext, Menge, Einheit, Preis, Rabatt, MwSt, **Foto**) und **Kalkulation** (Selbstkosten/Material/Arbeitszeit + Live-Summe/Marge über die zentrale Logik `lineNet`/`lineCost`/`vatTotals`). **Dokumentlokal** – ändert nur die Position dieses Dokuments, nie Leistungs-/Artikelstamm oder Kalkulation. Die Inline-Bearbeitung in der Tabelle bleibt zusätzlich bestehen. **Lang- und Textbaustein-Felder im Editor wachsen automatisch mit dem Inhalt** (zentrales `AutoGrowTextarea` in `DocumentCanvas.tsx`: Höhe = `scrollHeight`, `overflow-hidden`, kein internes Scrollen; misst auch den Initialwert via `useLayoutEffect`) – der Langtext ist damit immer vollständig sichtbar.

**Positions-Foto (dokumentlokal):** Beim Einsetzen einer Leistung/eines Artikels wird deren `image_url` als Snapshot in die Position (`DocPosition.image_url`) übernommen. In der Vollmaske kann das Foto angezeigt, **ersetzt/hochgeladen** (Upload in den mandantengetrennten Bucket **`document-images`**, Org-Ordner-Pfad, Migr. 0100) oder entfernt werden – ohne den Stamm zu ändern. Im PDF erscheint das Foto (größer, halbe Spaltenbreite) unter der Position **nur**, wenn die zentrale Darstellungsoption greift: `show_service_images` (Leistung) bzw. `show_article_images` (Artikel) aus `OfferDisplay` (Default **aus**). Technisch löst `resolvePositionImages` (in `printDocument.ts`) das Bild aus dem jeweiligen Bucket über eine signierte URL zu einer **base64-Data-URL** auf (wie das Logo), damit auch finalisierte PDF-Snapshots dauerhaft gültig bleiben (keine ablaufenden URLs). Bucket-Ermittlung: `detectBucket(image_url)` (Leistung→`service-images`, Artikel→`article-images`, dokumentlokaler Upload→`document-images`).

**Spaltenraster der Positionstabelle** (feste Prozentbreiten, Summe 100, `table-layout: fixed` für paged.js-Stabilität): Pos **4%**, Menge **7%**, Bezeichnung **descW** (= `100 − 4 − 7 − (Einheitspreis?11) − (Gesamt?11)`, also Standard **67%**), Einheitspreis **11%**, Gesamt **11%**. (Schmalere Pos/Menge/Geldspalten geben der Bezeichnung mehr Platz.) descW füllt den restlichen Platz, damit lange Bezeichnungen/Langtexte mehr Zeilenbreite bekommen; Einheitspreis/Gesamt bleiben rechtsbündig (`td.r { white-space: nowrap }`). `restoreColgroups()`/`repeatHeaders()` kopieren colgroup + thead auf Folgeseiten, damit die Spalten über alle Seiten fluchten. Spalten Einheitspreis/Gesamt werden je `OfferDisplay` ein-/ausgeblendet, descW wächst entsprechend.

`renderDocumentHtml(positions, summary, meta)` → HTML inkl. **paged.js** (`pagedjs@0.4.3`) → `POST /api/render-pdf` (PDFShift: `format:A4, margin:0, use_print:false, delay:1200, sandbox:false`) → echtes PDF im iframe-Viewer. Server-Key `PDFSHIFT_API_KEY` (nur serverseitig); 503 → Client-Fallback `printDocument()`.

**Download & Render-Fehler (Stand 2026-06-28):** Der App-„Herunterladen"-Button erzeugt programmatisch einen Anchor mit `download=<buildDocumentPdfFileName()>` und klickt ihn → garantiert immer der korrekte Dateiname (z. B. `ANGEBOT-0001-2026.pdf`), nie ein Blob-/UUID-Name. (Ausnahme: der **native** Chrome-PDF-Viewer-Download in der eingebetteten Leiste ignoriert unseren Namen – Chrome-Grenze, nicht steuerbar.) `htmlToPdfBlob()` liefert jetzt ein Ergebnis-Objekt `{ blob } | { error: { status, message } }`; `finish()`/`showPdf()` zeigen bei Render-Fehler eine **klare Meldung** im Viewer-Fenster (401/403 = Konfiguration, 429 = zu viele Anfragen, 408/504 = Timeout, 502/503 = Dienst, 0 = Verbindung) plus Buttons „Schließen"/„Stattdessen drucken" (Client-Fallback) – kein stilles Schließen mehr, das wie ein Internetproblem aussieht. `api/render-pdf.js` mappt PDFShift-Fehler auf saubere Status (401/429/502) **ohne** Secrets/Antworttext durchzureichen; Auth + Rate-Limit unverändert.

**`PrintMeta` (wichtige Felder)**: `docLabel, numberLabel, number, title, customer, date, vatLabel, recipientLines[], projectNumber, customerVatId, subtitleLines[], projectAddress, introHtml, prePositionsHtml, closingHtml, legalHtml, showLinePrices, display(OfferDisplay), payment(PaymentInfo), footerNote, showPageNumbers`.

**Seitenzahlen (Stand 2026-06-28):** „Seite X von Y" wird **immer** angezeigt (`counter(page)/counter(pages)` in `@bottom-right`). Das frühere Abschalten über `show_page_numbers` (Variante) wurde entfernt; die Spalte bleibt als no-op-Legacy bestehen und der Schalter ist aus dem Varianten-Dialog verschwunden.

**Einheitliche Schriftgröße der Rich-Text-Bausteine (Stand 2026-06-28):** Alle `.rt`-Blöcke (Vor-/Nachtext, „Einleitung vor Positionen" `.prepos.rt`, Rechts-/Zahlungstexte, Signatur) erzwingen per CSS `font-size: 10.5px !important; line-height: 1.5 !important` für `p, ul, ol, li, span, div, h1–h6`. Damit führen eingefügte/gemischte Formatierungen (Inline-`font-size` aus Copy&Paste, Überschriften) **nicht** mehr zu unterschiedlich großen Absätzen im PDF (Überschriften bleiben fett, aber in Textgröße). `!important` schlägt auch Inline-Styles aus DOMPurify (`USE_PROFILES:{html:true}` lässt `style` zu).

**Abschlussbereich & A4-Platznutzung (Stand 2026-06-28):** Nachtext und Signatur werden als **eine** Abschlussgruppe `<div class="closing-group">` mit `break-inside: avoid` gerendert → bleiben zusammen und füllen vorhandenen Platz nach der Zusammenfassung; bei echtem Platzmangel wandern sie gemeinsam auf die nächste Seite (keine halbe Leerseite nur mit Signatur). Der Zahlungsblock (`.paywrap`) bleibt außerhalb der Gruppe (steht davor), damit die Gruppe nicht zu groß wird; dadurch steht der **Nachtext jetzt hinter dem Zahlungsblock** (üblicher Abschluss-/Unterschriftsbereich). Der untere `@page`-Rand wurde von **28mm auf 24mm** reduziert (mehr Inhalt pro Seite); `@bottom-*`-Fußzeile + `padding-bottom:6mm` und die drucksichere Sichtbarkeit bleiben. `tr { break-inside: avoid }` (Position nie zerschneiden) + `tr.grp/tr.txt { break-after: avoid }` bleiben unverändert.

**Einfügen: Scroll + Aufleuchten (Stand 2026-07-07):** `useDocumentBuilder.lastInserted` trägt einen Modus – `append` (Plus/ans Ende → Scroll `block:"end"`, Position erscheint unten im Sichtbereich) vs. `insert` (Drag&Drop/gezielte Stelle → `block:"center"`). `DocumentCanvas` hebt die neue Position zusätzlich kurz hervor (CSS `.pos-flash` in `index.css`, ~1,5 s Akzent-Puls; `scroll-margin` an `[id^="pos-"]`). Mehrfach-Einfügen über Modals (Vorlagen/Zeiten/„Positionen einfügen") markiert per `builder.markInserted(letzteId, mode)` – „Am Ende" = `append`, gezieltes „Einfügen nach Position" = `insert` (mittig + Aufleuchten an der Einfügestelle). Gilt zentral für alle Hauptdokumente über den `DocumentWorkspace`.

**Editor-Drag&Drop (Stand 2026-06-28):** Collision-Strategie ist `pointerWithin` (Cursor muss wirklich über einer Dropzone sein). `computeDropIndex` liefert `number | null`; ein Sidebar-Element wird nur eingefügt, wenn beim Loslassen ein echtes Ziel getroffen wurde (`doc-end` oder eine vorhandene Position). Beim Zurückziehen außerhalb der Dokumentfläche: **kein** Einfügen, kein `bumpUsage`, kein Dirty, Drop-Indikator verschwindet. Sortieren bestehender Positionen + Plus-Quick-Add unverändert.

**Textdokument-PDF (Stand 2026-06-23, Block 6):** Dieselbe Engine (`buildHtml`) hat einen **Layout-Schalter** `PrintMeta.textMode` + `bodyHtml`. Im textMode entfallen Leistungstabelle, Zusammenfassung, MwSt und Zahlungsblock; stattdessen wird `bodyHtml` (Rich-Text) gerendert. Kopf/Logo/Empfänger/Betreff/Vor-/Nachtext/Fußzeile/Seitenzahlen/Signatur bleiben identisch (keine Doppellogik). Aufruf über `printTextDocument(meta)` (Fenster) bzw. `renderTextDocumentHtml(meta)` (Snapshot). Genutzt vom `TextDocumentEditor` für Dokumentarten mit `document_structure='text'`.

**Empfänger-Adressblock (`recipientLines[]`) – zentral, Stand 2026-06-26:** EINE Quelle: `contactRecipientLines()` in `src/lib/contact-name.ts`. **Jede Komponente auf eigener Zeile** (Reihenfolge: **Name/Firma → Zusatzzeile 1 (`recipient_extra_line1`) → Zusatzzeile 2 (`recipient_extra_line2`) → Straße → Adresszusatz (`address_extra`) → PLZ Ort**). Straße und Adresszusatz werden **NICHT mehr per Komma** zusammengeklebt (vorher „Straße, Adresszusatz" in einer Zeile → konnte Firma/Adresse verkleben). Alle Werte getrimmt, leere Zeilen entfallen. Die einzeilige **Betreff-Adresse** (Projekt/Kunde) kommt zentral aus `formatAddressInline()` (getrimmt → kein Leerzeichen vor dem Komma). Alle Editoren (Angebot/Auftrag/Rechnung/SUB; Nachtrag = Angebot) nutzen diese Funktionen – **keine** manuell zusammengebauten Empfänger-Arrays mehr.

**Gewerk-/Titelzeile im Layout (Stand 2026-06-26):** Nummer **und** Titel stehen in **einer** zusammengefassten Zelle (`grpTitleHtml(no, name)` → „Pos. N&nbsp;&nbsp;NAME"), nicht in der nur 4 % breiten Pos-Spalte – sonst lief „Pos. N" in den Titel der Nachbarspalte über (→ „Pos. 1GEMEINKOSTEN"). Die namenlose Sammelgruppe (keine Nummer) erscheint ohne „Pos."-Präfix (→ „Positionen" statt „Pos. Positionen"), auch in der Zusammenfassung. **Zahlungsblock (`payment`)** wird über `showPaymentForDoc(kind)` gesteuert: Angebot und Auftrag ohne (Stand 2026-07-06), Nachtrag/Rechnung/SUB mit.

**Spaltenraster**: Tabelle `table.lv { table-layout: fixed }` + `<colgroup>` in %: POS 6 %, Menge 10 %, Bezeichnung `descW = 100−6−10−EP−Gesamt`, Einheitspreis 12 %, Gesamt 14 % (Spalten je nach Darstellung). Zusammenfassungstabelle eigenes 6/80/14-Raster.

**Seitenumbruch-Fix „halb leere Seiten" (Stand 2026-07-07):** paged.js kopiert `<colgroup>` NICHT in Folgeseiten-Fragmente → die **Umbruch-Berechnung** rechnete dort mit gleich breiten Spalten (Bezeichnung schmal → Zeilen intern viel höher) und brach Seiten viel zu früh um; nach `restoreColgroups()` schrumpften die Zeilen zurück → riesige Leerräume (z. B. Seite 2 nur zu 1/3 gefüllt, Nachtext rutschte auf eine eigene Seite). Zentral behoben über **Zellbreiten-CSS** (`table.lv td.pos/qty/desc/r`, für die Zusammenfassung zusätzlich `td.zpos/znm` – bei `table-layout:fixed` definiert die erste Fragmentzeile die Spalten; `<col>` hat Vorrang, wo vorhanden); `restoreColgroups()`/`repeatHeaders()` bleiben als Sicherheitsnetz für die finale Optik. **WARNUNG (Bug 2026-07-07, nicht wiederholen):** KEIN `renderNode`-Hook zum Colgroup-Injizieren – der feuert schon beim Erst-Rendern, bevor paged.js die originale colgroup anhängt → doppelte colgroups → Seite 1/Zusammenfassung auf halbe Breite gequetscht. `restoreColgroups()` entfernt seither doppelte colgroups defensiv. Zusätzlich Abstände gestrafft: `.zus` 12px, `.paywrap` 14px, `.closing`/`.sign` 14px, `.block-post`/`.block-legal` 12px.

**Layout-Nacharbeit (Stand 2026-07-07, visuell mit paged.js-Screenshots verifiziert):**
- **Zusammenfassung NICHT mehr als Ganzes unteilbar:** `.zus` ohne `break-inside: avoid` – ein großer avoid-Block sprang bei vielen Titeln komplett auf die nächste Seite und ließ davor eine große Leerfläche. Jetzt fein gesteuert: `.zus-h { break-after: avoid }` (Überschrift klebt an der ersten Zeile), Titel-Summen-Tabelle bricht zwischen Zeilen (globales tr-avoid), nur der Summenblock `.tot` bleibt als Einheit zusammen.
- **Zusammenfassungs-Raster 10/76/14** (vorher 6/80/14): „Pos. 12" lief mit nowrap sonst optisch in den Titel („Pos.GEMEINKOSTEN").
- **Einheiten nie im Wort trennen:** `td.qty { overflow-wrap: normal }` + `.qu { hyphens: none }` (vorher `anywhere` → „paus/chal"); Umbruch nur zwischen Zahl und Einheit.
- **Kompakter, ruhiger:** `td` vertikal 5px (vorher 6px) → mehr Inhalt je Seite ohne Quetschen; `th` 7px + letter-spacing .4 → hochwertigerer Tabellenkopf.

**Zahlungsblock je Dokumenttyp (Stand 2026-07-06):** `showPaymentForDoc(kind)` (payment-conditions.ts): Angebot **und Auftrag** ohne automatische Zahlungsbox nach der Summe (Auftrag: Zahlungsbedingungen nur über Textbausteine/Platzhalter, z. B. `{{kondition.zahlungsbedingungen_text}}`); Nachtrag/Rechnung/SUB weiterhin mit Box.

**PDF-Cache zweistufig (Stand 2026-07-07):** `htmlToPdfBlob(html, cacheRef?)` (lib/pdf.ts).
- **Stufe 1 – Session-LRU:** fertige PDFs je SHA-256 des HTML (max 6, Speicher der Sitzung).
- **Stufe 2 – persistent (Migr. 0129):** privater, org-getrennter Bucket **`document-pdfs`** (Pfad `<org>/<source_table>/<source_id>/v<version_no>.pdf`) + Tabelle **`document_pdf_cache`** (`source_table, source_id, version_no, html_hash, storage_path`; unique je Quelle+Version; RLS `current_org_id()`). `version_no = 0` = Entwurf/Live (wird beim Neu-Rendern überschrieben), `> 0` = finalisierte Version (einmal erzeugt, stabil). **Bewusst eigene Tabelle statt Spalten an `document_versions`** – die Versionstabelle bleibt unveränderlich (keine UPDATE-Policy).
- **Gültigkeit:** Cache-Treffer NUR bei exakt gleichem `html_hash` (SHA-256 des kompletten HTML inkl. eingebettetem CSS) → Änderungen an Positionen/Texten/Preisen/Empfänger/Firma/Logo-URL/Layout ergeben einen neuen Hash; ein veraltetes PDF ist ausgeschlossen. Cache-Fehler sind nie blockierend (Fallback = normales Rendering).
- **Verdrahtung:** `DocumentWorkspace.doPdf` (Live = v0, gesperrt = Snapshot-Version), `VersionHistoryModal.openPdf` (je Version), Angebots-Mailversand (`OfferEditor`, v0 – Vorschau + Anhang teilen sich ein Rendering). `prepareDocumentPdf(ref, html)` rendert/persistiert im Hintergrund; `finalizeDocumentVersion` ruft es fire-and-forget nach jedem Abschluss mit `print_html` auf → das erste „PDF ansehen" einer finalen Version öffnet sofort.
- **Timing-Logs:** `console.debug("[pdf] …")` protokolliert HTML-Erzeugung, Cache-Lookup, PDFShift-Zeit und Blob-Größe (für Entwickler nachvollziehbar).

**Server-Rendering ohne Fix-Delay (Stand 2026-07-07):** `api/render-pdf.js` nutzt statt `delay: 1200` PDFShifts **`wait_for: "b4yPdfReady"`** – `printDocument.ts` definiert die globale Funktion (true sobald paged.js `__pagedReady` setzt, PDFShift-Cap 30 s). Kleine Dokumente rendern dadurch schneller, große warten exakt bis zur fertigen Paginierung (kein zu frühes Capturen). Für ÄLTERE gespeicherte Snapshots ohne `b4yPdfReady` injiziert der Server die Funktion defensiv selbst (plus 9 s-Sicherheitsdeckel gegen hängende Paginierung).

**Wichtige Lösungen (NICHT brechen)**
- `restoreColgroups()` (im `PrintHandler.afterRendered`): paged.js kopiert `<colgroup>` nicht auf Folgeseiten-Fragmente → per `data-ref` zurücksetzen → Spalten Seite 1/2/3 identisch.
- `repeatHeaders()`: setzt `<thead>` auf Folgeseiten mit **Overflow-Guard** (entfernt ihn, falls eine Zeile sonst abgeschnitten würde – nie Datenverlust).
- ESC-Viewer (`showPdf`): `b4yKey` per **Capture** auf window+document+Blob-iframe; nach Download/Druck Fokus zurück (`b4yRefocus`).
- **Rücksprung-Ziel (`returnUrl`)**: `openDocumentPdf`/`openSnapshotPdf` erfassen beim Öffnen automatisch `window.location.href` (App-Kontext) und reichen es als `returnUrl` an `showPdf`/`injectAppBar` durch (Param optional → Default = aktuelle URL). `b4yCloseView` schließt zuerst den Tab (Herkunftsfenster behält seinen Zustand, z. B. offene Versionshistorie); ist das Schließen blockiert, navigiert es exakt zur gespeicherten `returnUrl` (inkl. `?versions=1`), sonst Opener-URL/`document.referrer`/`/`. So funktioniert der Rückweg in beiden Fällen.

- **Genau eine Signatur:** Die Engine rendert die Grußformel/Signatur („Mit freundlichen Grüßen" + Firmen-Kontakt) **zentral**. Enthält der Nachtext (`closingHtml`) bereits „Mit freundlichen Grüßen", wird die Auto-Signatur **unterdrückt** → keine Doppel-Signatur. Standard-Nachtexte führen die Signatur daher nicht mehr selbst (Migration `0074`).

**Erweitern**
Layout/Spalten zentral in `buildHtml` (`<style>`, `COLGROUP`/`THEAD`). Neue Felder über `PrintMeta`. Gilt automatisch für ALLE Dokumenttypen (Angebot/Auftrag/SUB/Rechnung/Gutschrift/Mahnung/Nachtrag). Test mit Headless-Chrome + paged.js (Spaltenbreiten/Overflow messen).

**Menge/Einheit + Blocksatz (Stand 2026-06-27):** Die Menge-Spalte ist 10% breit (vorher 7%); Menge und Einheit liegen in `<td class="qty">` getrennt in `<span class="qn">` (Zahl, `nowrap`) + `<span class="qu">` (Einheit, umbruchfähig) – lange Einheiten wie „pauschal" laufen nicht mehr in die Bezeichnung und werden nie abgeschnitten. Kurz-/Langtext (`.nm`/`.lt`) sind im **Blocksatz** (`text-align: justify`); einzeilige Texte bleiben optisch links (justify streckt nie die letzte/einzige Zeile).

**Adresszusatz (Slash, Stand 2026-06-27):** Der Adresszusatz erscheint im Empfängerblock nicht mehr auf eigener Zeile, sondern als Slash-Suffix an der Straße: „Schrottgasse 7 / Stiege 2 / Top 4". Zentral in `src/lib/contact-name.ts` (`formatStreetLine`/`formatAddressExtraSuffix`); gespeicherte Werte (Komma) bleiben unverändert, nur die Anzeige wird normalisiert.

**Dokumentbezogene Empfängeranschrift (Override, Stand 2026-06-27):** Pro Dokument kann die Empfängeranschrift abweichend vom Kundenstamm gesetzt werden, ohne den Kontakt zu ändern. Speicherung als JSONB `recipient_override` am Beleg (offers/orders/invoices/sub_orders, Migration 0102): `{ enabled, name, line1, line2, street, address_extra, zip, city, country }`. Zentrale Auswahl `resolveRecipientLines(override, contact)` in `src/lib/contact-name.ts` (aktiver Override > Kundenstamm; Adresszusatz im Slash-Format, „Land" als letzte Zeile). UI: `RecipientOverrideEditor` in den Dokumenteinstellungen (Angebot/Auftrag/Rechnung). `meta.recipientLines` nutzt den Resolver → PDF + Versions-Snapshot frieren die dokumentbezogene Anschrift revisionssicher ein.

**Dokument-Signatur je `signature_source` (Stand 2026-06-28):** Die Signaturquelle wird **pro Dokument** gewählt und liegt als Spalte `signature_source text` (CHECK `company|creator|none`, Default `company`) auf `offers`/`orders`/`invoices`/`sub_orders` (Migration `0109`). `PrintMeta` trägt `signatureSource` (+ `createdBy`). Die async-Wrapper (`withSignature` in `printDocument.ts`) lösen die Quelle **zentral** über `resolveDocumentSignature({ source, createdBy, companyDefaultHtml })` aus `src/lib/document-signature.ts` auf:
- `none` → KEINE Signatur, **auch kein** Auto-Fallback (`meta._signatureNone = true` → `signatureBlock` liefert `""`).
- `company` → globale `company_settings.document_signature_html`; ist sie leer → automatische Firmen-Signatur (GF/Gesellschafter) als Notnagel.
- `creator` → persönliche `employees.document_signature_html` (via `auth_user_id = created_by`); leer → Firmensignatur; auch leer → Auto-Fallback.

Das frühere Mitarbeiter-Flag `document_signature_active` wird zur Auswahl **nicht mehr** ausgewertet (Spalte bleibt Legacy). `signatureBlock(meta, co)` rendert weiterhin **genau eine** Signatur: enthält der Nachtext bereits „Mit freundlichen Grüßen" → keine zusätzliche; sonst `meta.signatureHtml` (Grußformel nur vorangestellt, wenn nicht selbst enthalten); leer → Auto-Firmensignatur. UI: zentrale Komponente `SignatureSourcePicker` in den Dokumenteinstellungen (Angebot/Auftrag/Rechnung) und im SUB-Erstelldialog; zeigt schlicht, welche Signatur effektiv ins PDF kommt. Bewusst getrennt von der **E-Mail-Signatur** (`employees.signature_html`). Im finalisierten Snapshot fest eingebettet → revisionssicher (spätere Quellenänderung ändert ihn nicht).
